# Easing System

StateMotion replaces the hardcoded `smoothstep` progress easing with a configurable
easing evaluator shared by the C++ native effect and the TypeScript panel/host adapter.

## Why

The prior vertical slice baked `smoothstep` into `evaluateProgress`. Editors expect
Linear / Ease-in / Ease-out / Ease-in-out plus user-defined cubic-bezier curves
(like CSS `cubic-bezier`). Easing is a first-class, preset-storable transition property.

## Model

- `EasingMode` enum (permanent, append-only): `LINEAR=0`, `EASE_IN=1`, `EASE_OUT=2`,
  `EASE_IN_OUT=3`, `CUSTOM=4`.
- `EasingCurve { x1, y1, x2, y2 }`: canonical 0..1 control points. Named modes resolve
  internally to fixed curves; only `CUSTOM` reads the four stored points.
- Y is bounded to 0..1 (no overshoot) — a deliberate scope cut. Custom curves that would
  exceed 0..1 are clamped by the solver's endpoint handling.

## Contract

Inside the `progressCurve` disk-ID range (50–99), IDs 55–59 are reserved (confirmed free):

| diskId | logical ID            | native   | range | default            |
|--------|-----------------------|----------|-------|--------------------|
| 55     | `transition.easing`   | POPUP    | —     | 3 (EASE_IN_OUT)    |
| 56     | `transition.curveX1`  | FLOAT    | 0..1  | 1/3 (0.333…)        |
| 57     | `transition.curveY1`  | FLOAT    | 0..1  | 0.0                |
| 58     | `transition.curveX2`  | FLOAT    | 0..1  | 2/3 (0.666…)        |
| 59     | `transition.curveY2`  | FLOAT    | 0..1  | 1.0                |

`parameterCount` 20 → 25, `bindingRevision` 1 → 2. `oldProjectDefault` for the new params
is the legacy EASE_IN_OUT-equivalent (3 / 1/3 / 0.0 / 2/3 / 1.0). With control x-points
1/3 and 2/3, the cubic-bezier `BezierX(t) = t`, so `BezierY(t) = 3t²-2t³` — exactly the
prior `smoothstep` behavior. `0.42/0.58` is NOT the legacy equivalent and must not be used.

## Implementation

- `src/statemotion/progress/easing.{hpp,cpp,ts}` — single cubic-bezier solver:
  Newton-Raphson on `BezierX(t) - X` with analytic derivative, bisection fallback,
  non-finite input → 0, invalid control points → linear fallback, exact endpoints for all modes.
- `evaluateProgress` (`progress_engine.{h,cpp}` / `progressEngine.ts`) takes `easing` + `curve`
  in `ProgressInput`. Manual mode bypasses easing (eased == linear, deterministic midpoint).
  HoldA = 0 / HoldB = 1 are exact endpoints.
- Native `statemotion_effect.cpp` registers the five params from the binding table and passes
  the popup index + four control points into `buildProgressInput`.
- Panel `valueConversion.ts` treats `transition.easing` and `transition.curve*` as identity
  conversions (0..1, POPUP integer). `premiereAdapter` reads/writes them through the existing
  `LOGICAL_IDS` loop, so presets and live inspection carry easing automatically.

## Parity

`shared/fixtures/easing-fixtures.json` (generated by `tools/generate-easing-fixtures.js`) drives
both the C++ (`easing_test.cpp`, embedded JSON reader) and TS parity checks within `1e-6`.

## UI

The inspector shows a live **Easing** control for the selected clip: a curve selector
(Linear / EaseIn / EaseOut / EaseInOut / Custom) plus four curve inputs revealed only when
Custom is selected. Changes write through `writeLogical` to the effect.
