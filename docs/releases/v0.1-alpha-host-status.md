This Windows machine now has Visual Studio 2019 BuildTools (MSVC 14.29.30133,
toolset v142, cl 19.29.30159, x64). The SDK-free C++ suites were compiled and
executed for the first time.

- **SDK-free C++ compile:** PASS — all 11 SDK-free suites build + run, 0 failures
  (easing, progress_engine, transform_render, transform_state, native_adapter,
  host_time, render_input, integration, registration, identity, point_default).
  C++/TS easing parity confirmed against the shared fixture; CUSTOM `(1/3,0,2/3,1)`
  and named `EASE_IN_OUT` both equal `3x²-2x³` within 1e-6 in the compiled C++
  evaluator.
- **Easing type defect:** FIXED — `buildProgressInput` in `statemotion_host_time.hpp`
  previously typed its easing parameter as generated `ids::EasingMode`, but
  `ProgressInput.easing` uses `statemotion::EasingMode`; this caused real MSVC
  C2838/C2440 failures. Both enums share the same stable numeric identities 0–4,
  so the fix corrects the type boundary without changing runtime semantics.
- **Adobe SDK-dependent `statemotion_effect.cpp` compilation:** PENDING — its
  matching caller cast was corrected, but the translation unit requires the Adobe
  Effect SDK to compile and is not yet verified.
- **AEX Release x64 build:** BLOCKED — Adobe Effect SDK unavailable
  (`STATEMOTION_SDK` unset, no SDK headers on disk).

All native-host gates remain NOT YET VERIFIED.
